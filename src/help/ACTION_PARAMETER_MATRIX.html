Action Parameter Matrix

Scope

This guide is based on a debug-style scan of the action execution path across the project, not only the enum list.

Files reviewed for this guide include:

  • CommonPage.cs
  • CommonUtility.cs
  • WebElementUtility.cs
  • DataStore.cs
  • Table.cs
  • ListControl.cs
  • SelectControl.cs
  • DatePicker.cs
  • RadioControl.cs
  • AgGridControl.cs
  • CommonControls.cs
  • helper files under BromCom.Tests\Utilities\PageObjectModelHelper

Execution Path

The runtime flow is:

  1. CommonPage.CommonTestCase(...) iterates the parsed test steps.
  2. CommonPage branches special actions like SELECT, SWITCH_TAB, SWITCH_FRAME, FETCH_SHARED_STEPS.
  3. Most actions resolve an element through CommonUtility.GetElement(...).
  4. CommonUtility delegates element lookup to WebElementUtility.
  5. CommonUtility.ActionPerform(...) executes the action.
  6. Some actions delegate further into Table, ListControl, SelectControl, DatePicker, or a page-model helper.

Practical consequence:

  • a correct step is not just ActionType
  • it is ActionType plus the exact parameter shape expected by the implementation branch

Parameter Rules

Element

Usually the main UI locator.

But not always required. Some actions are data-only or browser-only:

  • DELAY
  • CHECK_URL
  • SAVE_HARDCODE_DATA
  • FETCH_SHARED_STEPS
  • alert actions

ElementCategory

This controls how Element is interpreted in WebElementUtility.

Supported categories:

  • XPATH
  • ID
  • TAGNAME
  • CSSSELECTOR
  • LINKTEXT
  • NAME
  • URL
  • JSPATH
  • VERIFY
  • VERIFYERROR

Important behaviors:

  • URL navigates using step.Value
  • JSPATH executes JavaScript and returns the element
  • VERIFY waits for an XPath element to exist
  • VERIFYERROR expects an XPath element and compares its text to step.Value

This section is the direct answer to “what do I put in Element, ElementCategory, and the related fields?”

The source of truth for this behavior is:

  • StepTemplate.cs
  • Constants.cs
  • WebElementUtility.cs

What to put in Element

Use these rules:

  • for XPATH, put the full XPath string
  • for ID, put the raw id only
  • for CSSSELECTOR, put the raw CSS selector only
  • for LINKTEXT, put the exact link text
  • for NAME, put the raw name value
  • for TAGNAME, put the raw tag name
  • for JSPATH, put a JavaScript expression such as document.querySelector(...)
  • for URL, do not think of Element as a locator; the navigation input is Value

Dynamic element variations

Dynamic element handling is implemented in WebElementUtility.cs.

There are two supported patterns.

Variation 1: Single replacement key with $$

Use:

  • IsElementPathDynamic = true
  • ElementReplaceTextDataKey = SomeSavedKey
  • Element = //div[text()='$$']

Runtime behavior:

  • the framework reads SomeSavedKey from DataStore
  • it replaces $$ with that value
  • then it resolves the final locator

Variation 2: Multiple replacement keys with Datakey1, Datakey2

Use:

  • IsElementPathDynamic = true
  • ElementReplaceTextDataKey = KeyA,KeyB
  • Element = //tr[td[text()='Datakey1']]/td[text()='Datakey2']

Runtime behavior:

  • the framework reads both keys from DataStore
  • Datakey1 is replaced with the first key value
  • Datakey2 is replaced with the second key value

Element authoring decision tree

Use this order:

  1. Decide whether the action needs element lookup at all.
  2. If it does, choose the locator category that matches the actual locator text.
  3. If the locator text changes at runtime, switch from static authoring to dynamic authoring.
  4. Only then fill Value, Key, and Headers based on the action branch, not the locator branch.

The important separation is:

  • Element and ElementCategory decide how the framework finds something
  • Value, Key, and Headers usually decide what the action does with that thing

Associated parameters that change element behavior

These fields are directly tied to the locator path, not just to the action:

Field Applies to Real effect
ElementReplaceTextDataKey dynamic element variations only picks which saved data replaces locator placeholders
IsElementPathDynamic dynamic element variations only tells WebElementUtility not to use Element as-is
Value URL, VERIFYERROR, plus the action itself can be part of locator behavior for some categories
Key any action that reads from DataStore does not change lookup, but often changes what gets typed/selected after lookup
Headers helper or structured actions does not change lookup, but can change how the located element is interpreted later

Invalid combinations to avoid

Combination Why it fails
Element has XPath syntax but ElementCategory=CSSSELECTOR wrong Selenium lookup method is used
ElementCategory=ID with Element=#myId By.Id expects raw id only
empty ElementCategory on an action that still performs element lookup WebElementUtility falls into the invalid-category branch
IsElementPathDynamic=true with empty ElementReplaceTextDataKey dynamic replacement has no source key
dynamic locator without $$ or Datakey1 tokens replacement flag is set but the locator template has nothing to replace
ElementCategory=URL with the real URL stored in Element but empty Value navigation branch reads Value, not Element
ElementCategory=VERIFYERROR without Value the compare target is missing

When Element can be omitted

You usually do not need Element for:

  • browser-only actions like OPEN_NEW_TAB, CLOSE_TAB, REFRESH_PAGE
  • data-only actions like SAVE_HARDCODE_DATA
  • comparison actions that operate entirely on DataStore
  • alert-only actions like VISIBLE_ALERT
  • CHECK_URL
  • TRIGGER_SQL_QUERY

Element authoring mistakes to avoid

  • using XPATH but providing CSS selector syntax
  • using ID and including #
  • setting IsElementPathDynamic = true without providing ElementReplaceTextDataKey
  • using $$ in Element without saving the replacement key first
  • treating URL as a normal locator category
  • assuming VERIFY returns an element for later logic; it is mostly a wait/checkpoint path

Recommended element patterns

Prefer:

  • stable ids
  • stable names
  • short XPath with user-visible anchors
  • saved-value dynamic locators only when necessary

Avoid:

  • long position-based XPath if a stable attribute exists
  • brittle text nodes that change with formatting
  • dynamic locator templates when a fixed locator is enough

ElementReplaceTextDataKey

This only matters when IsElementPathDynamic is true.

Observed behavior in WebElementUtility.cs:

  • if ElementReplaceTextDataKey contains one key, the framework replaces $$ inside Element with the stored value
  • if ElementReplaceTextDataKey contains multiple comma-separated keys, the framework uses CommonControls.AddDatakeyValueToString(...) and replaces Datakey1, Datakey2, and so on

This is a missing-but-important authoring rule for dynamic locators.

IsElementPathDynamic

When true, Element is not treated as a final locator string. It is treated as a template that must be resolved through ElementReplaceTextDataKey.

Description

Description is parsed from Azure DevOps and available on the step model. It is not one of the main dispatch keys, but some specialized logic may still depend on it. Do not assume it is disposable.

StepDescription

StepDescription is built from the full step text and is used in failure reporting via ExceptionHandler.cs. This is the human-readable diagnostic string a QA will see on failure.

Value

Value is the most overloaded field after Headers.

Common meanings:

  • typed text
  • a dropdown display text
  • a row count
  • a day offset
  • a boolean-like control string
  • a dictionary-style config payload
  • a comma-separated multi-argument payload

ExpectedValue

Common meanings:

  • exact expected text
  • true or false
  • comparison mode
  • output format string
  • saved key list for XML helpers

Key

Key is used with DataStore, which is browser-specific.

Key can mean:

  • read input from stored data
  • write output to stored data
  • compare current UI against stored data

Headers

Headers is a generic extra-parameter field. It is not just a table header.

It can mean:

  • split mode
  • date format
  • arithmetic mode
  • partial match switch
  • xpath
  • table column name
  • helper-specific mini configuration

IsConcatenated

This is a real runtime flag, not just metadata.

Observed use:

  • concatenated compare flows like CONTAINS
  • tooltip/table comparison flows

If a step relies on combining values from multiple headers into one compare string, IsConcatenated can be required.

Dictionary-Style Value

Many advanced actions use Value in this form:

rownumber:1,columnnumber:3

or:

take:5,skip:0

Parsed by:

  • CommonUtility.GetValuesDictionary(...)

Observed rules:

  • keys are lowercased
  • most values are lowercased
  • values for keys containing xpath, possibletext, or headercount preserve case

Safe keys seen in code:

  • rownumber
  • columnnumber
  • skip
  • take
  • style
  • attribute
  • iconnumber
  • tableheaderxpath
  • headertablexpath
  • headercount
  • possibletext
  • tag
  • divnumber

  • Value=

This matters because a field missing in Azure DevOps does not exist later in the runtime model.

DataStore Rules

The framework stores values per browser in DataStore.

That means:

  • a Key saved in Chrome is not shared with Firefox
  • TYPE, SELECT, VERIFYDATA, and many comparison actions can fail if the prerequisite save step ran in a different browser context

Delegated Business Actions

These actions are implemented in helper classes and are not safe to author from scratch unless you copy an existing working pattern.

Communication Helper Actions

Defined in:

  • CommunicationModelHelper.cs

Observed contracts:

  • ADD_EDIT_PHONE_EMAIL Key acts like a mode switch: email or phone Value is the email address or phone number to enter
  • OVERVIEW_EMAIL_SENT_VERIFY Value is the message text Headers is a comma-separated list of table columns
  • SCHEDULE_EMAIL_SUBJECT_BODY_CHECK Value is subject,body Headers is the relevant table column pair
  • MATCH_RECIPIENT_DATA_BY_STUDENT Key references saved table data Headers selects the header mapping Value is the expected message type

Group Helper Actions

Defined in:

  • GroupsModelHelper.cs

Observed contracts:

  • ADD_SCHEDULE_DATA_IF_NOT_EXIST Headers is used as an XPath probe to decide whether data already exists
  • ADD_GROUP_DETAIL_DATA_IF_NOT_EXIST Value controls count or true ExpectedValue may short-circuit by current column count Headers becomes a comma-separated mini script containing tokens such as TYPE, STYPE, DATEPICK
  • ADD_BAND_DATA, ADD_INTERVENTION_DATA, ADD_WITHDRAWAL_DATA no stable user-authored parameter contract beyond the current page state

Payment Helper Actions

Defined in:

  • PaymentModelHelper.cs

Observed contracts:

  • ADD_ORDER_IF_NOT_PRESENT Value or Key supplies the student selector text
  • ADD_OR_TRANSFER_BALANCE_TO_MAKE_POSITIVE Value is the target balance
  • RELOGIN_INTO_SAGEPAY_IF_LOGGEDOUT current element/page state drives the behavior; do not author freehand
  • TRANSFER_BALANCE_TO_SIBLING, REMOVE_TRANSACTION_TO_MAKE_BALANCE_NEGATIVE driven mostly by saved data and fixed helper logic

Student and Staff Deleted-Item Actions

Defined in:

  • StudentModelHelper.cs
  • StaffModelHelper.cs

Observed contracts:

  • Key usually points to previously saved table data
  • Headers usually names one or more table columns
  • Value often controls sort direction or expected matching values

These are safe to reuse only by copying a nearby deleted-item workflow.

Timetable / Missing Register Helper Action

Authoring Rules From The Scan

  1. If an action is implemented directly in CommonUtility, it is usually safe to document and author with a controlled pattern.
  2. If an action immediately delegates to a helper file, do not invent a new parameter pattern. Copy a working existing test.
  3. If Value contains commas and colons, assume it is a dictionary-style config payload unless you verified otherwise.
  4. If Key is present, always check whether the action reads from DataStore instead of using Value.
  5. If Headers is present, do not assume it is a literal header label. It may be a mode string, split token, xpath, or mini DSL.

Safe QA Strategy

For a new step:

  1. Find the exact action branch in CommonUtility.cs.
  2. Check whether it delegates to Table, ListControl, SelectControl, DatePicker, or a page-model helper.
  3. Match the parameter shape from a real working test in the same module.
  4. Only generalize the step if the parameter contract is clearly stable in code.
